Built for Reliability

Every component designed for distributed systems that can't afford to lose a job.

Distributed Consensus

Raft-based strong consistency with automatic leader election and log replication. No split-brain, ever.

Two Job Types

Unico for one-time execution at a specific timestamp. Recurrente for cron-based recurring schedules with optional end dates.

Time-Slotted Queue

O(1) slot lookup via cached min key. Hierarchical Timing Wheel (seconds, hours, days) for long-range scheduling.

Capacity Management

Memory and job count limits prevent OOM. Cold spilling to BoltDB when capacity is constrained. Hot slots stay in memory.

Split-Brain Prevention

RA-style minority partition shutdown. Graceful leader resignation on shutdown. Automatic failover when leaders die.

Observability

Prometheus metrics, OpenTelemetry integration, three-tier health status, and structured logging. Grafana-ready out of the box.

Architecture

Three nodes, one leader, Raft consensus. Every state change is replicated before it's committed.

How It Works

Three steps from job submission to execution. Raft handles the hard parts.

1

Submit

Send a job via the REST API. Define either a one-time timestamp or a cron expression.

# Create a one-time job
curl -X POST https://cluster:8080/jobs \
  -d '{"type":"unico","payload":"...","run_at":"2025-06-01T12:00:00Z"}'
2

Consensus

The leader replicates the job through the Raft log. A majority must acknowledge before the FSM applies the state.

// Raft log replication
leader -> append_entries -> followers
majority ? commit : retry
// FSM applies state
jobToSlot[job.ID] = slot
3

Execute

The worker picks up the slot at the right time, dispatches the webhook, and tracks status through the full lifecycle.

pending → in_progress
→ completed | failed
→ timeout | cancelled
// Idempotent: checks status first

Job Types

Two scheduling modes designed for every scenario. Intentionally Spanish names — they stay.

Unico

One-time execution

Fire-and-forget scheduling. Set a specific timestamp, and the cluster guarantees execution — exactly once, even on failover.

// Example Unico job
{
  "type": "unico",
  "payload": "send-alert",
  "run_at": "2025-06-01T12:00:00Z",
  "webhook_url": "https://..."
}
Fire & Forget

Recurrente

Cron-based recurring

Recurring execution using standard cron expressions. Add an optional end date to gracefully terminate the schedule.

// Example Recurrente job
{
  "type": "recurrente",
  "payload": "cleanup-db",
  "cron_expr": "0 3 * * *",
  "last_date": "2025-12-31T23:59:59Z",
  "webhook_url": "https://..."
}
Cron-Based

Quick Start

From zero to a running 3-node cluster in under a minute.

1

Launch the Cluster

# Clone and start a 3-node cluster
git clone https://github.com/manudiv16/scheduled-db
cd scheduled-db
make dev-up

# Check cluster health
curl https://localhost:8080/health
2

Create Jobs

# Create a Unico (one-time) job
curl -X POST https://localhost:8080/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "type": "unico",
    "payload": "hello-world",
    "run_at": "2025-06-01T12:00:00Z",
    "webhook_url": "https://example.com/hook"
  }'

# Create a Recurrente (recurring) job
curl -X POST https://localhost:8080/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "type": "recurrente",
    "payload": "cleanup",
    "cron_expr": "0 3 * * *",
    "webhook_url": "https://example.com/hook"
  }'
3

Check Status

# Get job status
curl https://localhost:8080/jobs/{id}/status

# View execution history
curl https://localhost:8080/jobs/{id}/executions

# Check cluster info
curl https://localhost:8080/debug/cluster

API Reference

Clean REST endpoints for complete job lifecycle management.

POST /jobs Create a new job
GET /jobs/{id} Get job details
DELETE /jobs/{id} Delete a job
GET /jobs/{id}/status Get job status
GET /jobs/{id}/executions Get execution history
POST /jobs/{id}/cancel Cancel a job
GET /health Health check
GET /debug/cluster Cluster info
